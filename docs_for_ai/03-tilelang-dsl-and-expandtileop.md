# 03 — `tilelang-dsl`, `lib/TileOps` templates & `ExpandTileOp`

This is the **legacy, compiler-internal** DSL path — the thing #739 wants to fold into `ptodsl`.
It answers one question for the compiler: *given a concrete `pto.tadd` (etc.) TileOp, what
micro-instruction body implements it for this dtype/shape/layout?* The answer is generated by a
Python template, fetched at compile time over a socket.

```
ExpandTileOp (C++ pass) ──spec──▶ Tilelang daemon (Python subprocess) ──▶ tilelang-dsl
        ▲                                                                       │
        └──────────────── specialized MLIR func (parsed + inlined) ◀───────────┘
```

## 1. `tilelang-dsl` — the template DSL

Location: `tilelang-dsl/python/tilelang_dsl/` (18 modules) + docs/examples/tests.
It is a standalone package; `tilelang-dsl/` owns the v1 core, while legacy
`python/pto/dialects/pto.py` keeps the older experimental VPTO surface (README.md:6–14).

**Template author API** — `@pto.vkernel(...)` (`kernel.py:980–1010`):
```python
@pto.vkernel(target="a5", op="pto.tadd")           # target: only "a5" today
def template_tadd(src0: pto.Tile, src1: pto.Tile, dst: pto.Tile):
    dtype = dst.element_type
    valid_rows, valid_cols = dst.valid_shape
    for row in range(0, valid_rows, 1):
        remained = valid_cols
        for col in range(0, valid_cols, pto.get_lanes(dtype)):
            mask, remained = pto.make_mask(dtype, remained)
            lhs = pto.vlds(src0[row, col:])
            rhs = pto.vlds(src1[row, col:])
            pto.vsts(pto.vadd(lhs, rhs, mask), dst[row, col:], mask)
```
Decorator metadata (advanced/matcher mode): `dtypes=[...]` (multiple polymorphic signatures with
`AnyFloat`/`TypeVar` wildcards), `constraints=[...]` (callable predicates), `priority=N`,
`ops=[...]` (multi-op). Tile API: `tile[row, col:]`, `.element_type`, `.valid_shape`,
`pto.vlds/vsts`, `pto.vadd/vsub/vmul/vmax/...`, `pto.make_mask`, `pto.get_lanes`,
plus advanced `pto.strict_vecscope`, raw `ptr(...)`, `copy_*` DMA.

Specialization (`v1-surface.md:131–159`): `kernel.specialize(dst=pto.TileSpecialization(shape=(16,32),
memory_space=..., config=...), ...).mlir_text()` produces the concrete MLIR text.

## 2. The template library — `lib/TileOps/*.py`

~85 `*_template.py` files (one per TileOp family): `tadd_template.py`, `tsub_template.py`,
`tmul_template.py`, `tmatmul_template.py` (uses `@pto.ckernel` + `pto.mad`), reductions
`tcolmax_template.py`/`tcolmin_template.py`, transcendentals `texp_template.py`/`tlog_template.py`,
etc. Templates with **constraints** validate context (e.g. `tcolmax_template.py:13–38` checks
layout/`valid_shape`).

`lib/TileOps/render_template_mlir.py` (284 lines) is a standalone CLI to materialize a template to
MLIR for inspection/testing — handy for debugging without running the full compiler:
```bash
python3 lib/TileOps/render_template_mlir.py lib/TileOps/tadd_template.py \
  --tile dst=8x64@ub --tile src0=8x64@ub --tile src1=8x64@ub -o /tmp/tadd.mlir
```
It dynamically imports the template, finds `@pto.vkernel` descriptors, binds concrete op/dtype
(wildcards → defaults, e.g. `AnyFloat`→`f32`), and caches by
`(op_name, target, operand_types, operand_shapes, tile_config)`.

## 3. `ExpandTileOp` — the C++ pass

`lib/PTO/Transforms/ExpandTileOp.cpp` (~44 KB). Workflow (header comment, lines ~12–25):
1. Extract a `SpecKey` from **all** operands' `tile_buf` types (`SpecKey` at ~143–153:
   opName, targetArch, per-operand `OperandTypeInfo` at ~97–138 = dtype/shape/valid_shape/
   memory_space/layout/config, plus context attrs).
2. Invoke the Python DSL (via the daemon) to generate a specialized MLIR function with `tile_buf` params.
3. Parse the returned MLIR and clone the function into the module.
4. Replace the `pto.t*` op with a `func.call` passing the tile_buf operands.

Daemon invocation (`ExpandTileOp.cpp:774–868`): locate `python3`, build operand-specs + context-attrs
JSON, run `python3 -m tilelang_dsl.daemon_helper --socket … --target a5 --op pto.tadd
--operand-specs '[…]' --context-attrs '{…}'` (30 s timeout), read MLIR from a temp file, parse + clone
(~871–911). **Context attrs** carried into the template (~407–447): `round_mode` (TCvt), `cmp_mode`
(TCmp), `precisionType` (TExp/TLog/TSqrt/TRecip/TRsqrt/TDiv: Default vs HighPrecision), `rounds` (TRandom).

Companion passes: `pto-fold-tile-buf-intrinsics` (`FoldTileBufIntrinsics.cpp`) folds structured-view
intrinsics after inlining; `pto-inline-libcall` (`PTOInstantiateAndInlineOpLib.cpp`) materializes and
inlines OP-Lib instance bodies; `pto-lower-to-oplib-calls` (`PTOLowerToOpLibCalls.cpp`).

## 4. The Tilelang daemon

So the compiler doesn't pay Python cold-start per TileOp, `ptoas` runs a long-lived Python server.

- **C++ side** (`tools/ptoas/TilelangDaemon.{h,cpp}`): `DaemonManager::start/stop/isRunning`
  spawns `python3 -m tilelang_dsl.daemon --socket /tmp/tilelang_daemon_<pid>.sock --template-dir
  lib/TileOps`, sets `PYTHONPATH`, registers an `atexit` cleanup (SIGTERM + socket unlink).
- **Python side** (`tilelang_dsl/daemon.py`, `daemon_helper.py`, `daemon_core.py`, `daemon_client.py`):
  `daemon_core.py` holds an in-memory FIFO `InstanceCache` (max 1000) with single-flight dedup of
  concurrent identical requests; `instantiate()` (~79–224) computes a stable key, checks cache, filters
  descriptors, selects a kernel, builds tile bindings from operand specs, specializes, and returns
  `specialized.mlir_text()`.

## 5. Version selection — how it works *today*

In `daemon_core.py` descriptor filtering (~398–469), in order:
1. target match (`descriptor.target == target`)
2. op match (`op_name in descriptor.match_ops`)
3. parameter-kind match (tile/view/scalar align with operand specs)
4. dtype-signature match over `descriptor.dtypes`, wildcard/`TypeVar` aware (`_dtype_matches`)
5. constraint predicates evaluated (`select_kernel` in `kernel.py`)
6. priority tie-break (highest `priority` wins; remaining ties → error)

So selection today is **deterministic and rule-based**, scoped to the *template/specialization*
level — there is no cost model, and the deeper "which C++ implementation variant" choice
(1D vs 2D, post-update vs not) happens later, inside `pto-isa` template instantiation, invisible to
this layer (see [05-pto-isa-library.md](05-pto-isa-library.md)). This is exactly the layering #739
wants to lift up so the choice becomes MLIR-visible.

## 6. Current status & relationship to `ptodsl`

- ✅ This path is the **live, working** TileOp-expansion mechanism today (85 templates, daemon, caching).
- ⚠️ **Duplication is the problem, not bugs.** `tilelang-dsl` and `ptodsl` are parallel codebases with
  similar surfaces (`tile[row, col:]`, simd/cube notions) that must be changed in lockstep. No
  TODO/FIXME/DEPRECATED markers in templates/daemon/ExpandTileOp — the migration is tracked in #739,
  not in code comments.
- ❌ **No migration code exists yet.** Per Alan Wang (China team) there is no proposal wiring `ptodsl`
  into `ExpandTileOp`; it must be designed (registration, matching, constraints, version selection,
  instantiation interface, optimization pipeline). The design discussion is captured in
  [ptodsl_tilelang_expandtileop_summary.md](ptodsl_tilelang_expandtileop_summary.md).

Evolution of `tilelang-dsl` itself (`docs/matcher-and-advanced-surface-migration.md`):
v1 core (monomorphic) → v1 lowering (DMA/vector/loops) → matcher + advanced surface (multi-signature,
constraints, priority, raw pointers, low-level DMA) → template-op-slots. Deferred: some reduction-family
and advanced vector ops not yet exposed in authoring form (`v1-lowering.md:105–123`).
